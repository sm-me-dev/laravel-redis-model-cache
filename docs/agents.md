## Package Architecture & Testbench

This is a Laravel package using **Orchestra Testbench** to simulate a booted Laravel application sandbox for feature and integration tests.

### Test Case & Bootstrapping
- **Base Test Class:** All tests extend `Orchestra\Testbench\TestCase` (defined in [tests/TestCase.php](file:///home/sm-me/Codes/Laravel/laravel-redis-model-cache/tests/TestCase.php)).
- **Service Providers:** Registered via `getPackageProviders($app)` on the test class.
- **Environment Setup:** Set config/env settings inside `getEnvironmentSetUp($app)`.
- **Database Migrations:**
  - Load migrations automatically with the `RefreshDatabase` trait.
  - Override `defineDatabaseMigrations()` to call `$this->loadMigrationsFrom(...)` for custom database schema paths.
  - Load core Laravel migrations with `$this->loadLaravelMigrations()`.
- **PHP Attributes:** Use `#[WithEnv('KEY', 'value')]`, `#[WithConfig('config.key', 'value')]`, or custom environment method execution using `#[DefineEnvironment('methodName')]`.

### Testbench CLI Commands
- Run migrations: `vendor/bin/testbench migrate`
- Interactive tinker sandbox: `vendor/bin/testbench tinker`
- Run seeds: `vendor/bin/testbench db:seed`
- Purge skeleton: `vendor/bin/testbench package:purge-skeleton` (or `composer clear`)
- Package discovery: `vendor/bin/testbench package:discover` (or `composer prepare`)
- Run tests: `vendor/bin/phpunit` or `vendor/bin/pest` (or `composer test`)

### Workbench Directory (`workbench/`)
- The `workbench/` directory is generated by Testbench. Do not operate/modify files directly inside it as sandbox purges will clean and regenerate it.
- Autoloading namespaces for sandbox-only code are defined in `composer.json` under `autoload-dev`:
  - `Workbench\App\` maps to `workbench/app/`
  - `Workbench\Database\Factories\` maps to `workbench/database/factories/`
  - `Workbench\Database\Seeders\` maps to `workbench/database/seeders/`

## Feature Areas
- Redis hash-based model caching with index/sorted-set lookups
- Stale-while-revalidate (SWR) pattern with background queue jobs
- Stampede protection with Redis lock mechanism
- Compression (zstd/gzip) support
- Pluck/hydrate with configurable batch size (uses HMGET for O(1) round trips)
- `selective()` method for lightweight field-only fetches (60-80% less memory)
- Compression min_size threshold to skip CPU waste on small payloads
- Incremental index updates on model changes
- Debug tooling (inspect, analyzeIndexes, debug mode)
- Telescope & Pulse integration (optional)
- Observability metrics (hit rates, latency percentiles)
- Warmup artisan command
- Benchmark suite (in `benchmarks/`) for throughput, latency, memory
- Multi-tenant cache isolation via `{tenant:{id}:{table}}:` key prefixing
- `TenantResolverInterface` with default `RequestTenantResolver` (header/subdomain/auth/session strategies)
